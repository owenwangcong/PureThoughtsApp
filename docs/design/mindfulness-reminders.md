# 设计文档 · 正念提醒(定时定点发声)

> 对应 PRD §9.3「正念提醒」的详细设计。状态:**设计定稿(开放问题已全部落定),待批准实现**。
> 作者:Claude(2026-07-14) · 决策依据:用户十点定案(见 §1),开放问题见 §15。

## 1. 需求与已定决策

用户希望:可设置**周几**、**几点到几点**之间、**每隔多少分钟**响一声磬,提醒自己保持正念;
即便 App 在后台、息屏、甚至被关闭也要能响。

产品决策(2026-07-14 用户拍板):

1. **尊重系统静音**——手机处于静音 / 勿扰(DND)时不强行出声,仅留通知;回到响铃模式自动出声。
   (iOS 本就无法穿透静音;此决策让两端行为一致、且商店合规。)
2. **呈现形式 = 通知横幅 + 磬声**——这是 App 关闭时唯一能发声的形式。
3. **默认周几 = 全选七天**,用户可改(§7)。
4. **间隔下限 = 10 分钟**(§6 模型;更小会让 iOS 迅速超 64 且过扰)。
5. **不支持跨午夜时间窗**(2026-07-14 复议:同日窗口更好懂、代码更简、正念场景本就是白天)——窗口必须同日,结束时间须晚于开始时间(§4.1)。
6. **单套时间表**(MVP 不做工作日/周末分设)。
7. **通知文案用户自定**(一段文字,默认给占位文案)。
8. **需要「设置帮助」界面**,分别指引 iOS 与 Android 如何授权 / 放行(§9)。
9. **双端(iOS/Android)体验一致**:同一套 UI / 文案 / 磬声 / 行为;仅"系统强加的差异"(iOS 64 上限、无法穿透静音)以提示形式说明,不做成两套功能(§10.1)。
10. **本次只出设计文档**,实现时机与范围后续再定。

## 2. 核心技术原理(为什么这样做)

**关键认知:App 被关闭后无法"跑一个计时器"。** iOS 完全不允许后台定时执行;
大陆安卓的常驻后台会被厂商杀。唯一能在 App 不运行时于预定时刻发声的机制,是
**提前把每一次响铃注册成操作系统级的本地通知**,由 OS 到点触发。

因此本功能的本质是:**把"周几 × 时间窗 × 间隔"展开成一组固定的"每周重复"通知槽,
一次性注册给系统**,而不是维护一个运行中的队列。这样:

- App 关闭 / 重启 / 息屏都不影响——通知由系统持有并触发;
- 不需要服务器,不用 FCM/APNs(FCM 在大陆是死的),纯设备本地;
- 复用现成磬声 `assets/audio/bell.wav`。

**推送(FCM/APNs)是错误工具**:要服务端、FCM 大陆不可达、且保证不了分钟级精度——本功能一律不走推送。

## 3. 平台能力边界(硬天花板,不可逾越)

| 能力 | iOS | 安卓(海外) | 安卓(大陆) |
|---|---|---|---|
| 关 App / 息屏也能响 | ✅ 本地通知照常触发 | ✅ | ⚠️ 依赖用户手动开自启动+电池不限制,否则被厂商杀 |
| 穿透"静音/勿扰" | ❌ 需 Apple「Critical Alerts」权限(仅批医疗/安全类) | ⚠️ 可用闹钟音量,仍受勿扰约束 | ⚠️ 同左 |
| 预定通知数量上限 | ⚠️ **系统最多 64 条待触发** | 基本无限 | 基本无限 |
| 无通知的纯后台响铃 | ❌ 不可能 | ⚠️ 需常驻前台服务 | ❌ 最易被杀 |

**结论**:所谓"权限越高越好"有硬顶——iOS 无法穿透静音、无法超过 64 条待触发;
大陆安卓无法从代码层面阻止厂商杀进程。我们能做的是"把该要的权限都要上 + 引导用户手动放行",
不能突破上述边界。设计必须诚实地把这些边界暴露给用户,而不是假装能做到。

## 4. 调度模型:每周重复的"通知槽"

把配置展开为一组 **(周几, 时:分)** 槽位,每个槽位注册为一条**按周重复**的本地通知:

```
槽位集合 = { (weekday, t) | weekday ∈ 选中的周几,
             t ∈ [start, start+interval, start+2·interval, …] 且 t ≤ end }
槽位数   = |选中周几| × floor((end − start) / interval + 1)
```

- **安卓**:`flutter_local_notifications.zonedSchedule(..., matchDateTimeComponents: DateTimeComponents.dayOfWeekAndTime)`,
  每槽一条按周重复;数量无硬上限;插件自带 `BOOT_COMPLETED` 接收器,重启后自动恢复。
- **iOS**:`UNCalendarNotificationTrigger(repeats: true)`,每槽一条按周重复;
  **受 64 条系统上限约束**。只要槽位数 ≤ 64,**注册一次后永久生效,App 永不用再打开也会响**。

> 这是本设计的关键优势:用"每周重复槽"而非"滚动队列",让"关了 App 也能永远响"在两端都成立
> (iOS 以 64 槽为限)。只有 iOS 槽位 > 64 时才需要降级(见 §5)。

### 4.1 时间窗必须同日(不跨午夜)

- **约束**:结束时间必须**晚于**开始时间(`start < end`),窗口落在同一自然日内。
- **校验**:`end ≤ start` 时设置页拦截并提示"结束时间需晚于开始时间";最宽窗口 00:00–23:50。
- **好处**:所有槽位都落在所选 weekday 当天,展开逻辑简单(无 24:00 回绕),
  "本周共 M 次"计数直白,规避周日→周一回绕与 `end==start` 歧义;对年长用户也更好懂。
- **边缘取舍**:无法设置真正跨零点的窗口(如 22:00→次日 02:00);此类极少见,
  可用同日宽窗口近似,不为其增加模型复杂度(YAGNI)。

### 时区与夏令时
- 一律按**设备本地时区**注册(`timezone` 包的 local location)。
- 用户跨时区旅行:时间随本地钟走(如设 8:00,到新时区仍是当地 8:00)——符合"正念作息"直觉。
- 夏令时切换由系统日历触发处理;按周重复的日历触发天然跟随。

## 5. iOS 64 槽超限的处理

槽位数 = 周几数 × 每日次数。示例:每 30 分钟、9:00–18:00(19 次/天)× 7 天 = 133 > 64。

设置页**实时计算并显示槽位数**,超 64 时:

- **首选**:提示用户调整(增大间隔 / 缩短时间窗 / 减少天数),并给出"当前 N 槽 / 上限 64"的即时反馈
  及一键建议(如"改成每 60 分钟即可 ≤64")。
- **可选降级(需明确告知)**:"滚动模式"——只排最近 64 次,App 每次打开时补排后续。
  代价:**若连续多日不打开 App,iOS 上会在排满的 64 次后断响**,直到重开 App。
  仅在用户理解并接受后开启。

安卓无此限制,同一配置照常全部注册。

## 6. 数据模型(本地存储,不入云)

存于设备本地(`shared_preferences` JSON,或后续 `sqflite`;**不同步到 Supabase**——纯个人设备级偏好):

```dart
class MindfulnessSchedule {
  bool     enabled;          // 总开关
  Set<int> weekdays;         // 1=周一 … 7=周日(ISO-8601)
  int      startMinutes;     // 自 00:00 起的分钟(如 8:00 → 480)
  int      endMinutes;       // 结束(> startMinutes)
  int      intervalMinutes;  // 间隔(预设 15/20/30/45/60/90;最小 10)
  String   sound;            // 'bell'(默认磬声)| 'silent'(仅震动)
  bool     vibrate;          // 是否震动
  String?  message;          // 通知文案(默认"该正念了 · 回到当下")
}
```

不做跨设备同步的理由:提醒是设备级行为(通知在设备上触发),换机重设成本低,
避免为一个本地功能引入云端表 + RLS + 冲突合并的复杂度(YAGNI)。若日后有强需求再加。

## 7. 用户交互设计

**入口**:工具页(与打坐計時 / 念珠計數同组)新增「正念提醒」;或设置页。

**设置页**:
- 总开关(关闭即取消全部已注册通知);
- **周几**:七个可多选 chip,**默认全选七天**,用户可改;
- **时间窗**:起 / 止时间选择器(`showTimePicker`);同日窗口,结束须晚于开始(§4.1);
- **间隔**:步进 / 预设(10/15/20/30/45/60/90 分钟),**下限 10 分钟**;
- **通知文案**:单行输入,用户自定;默认占位"该正念了 · 回到当下"(简繁各一份默认);
- **铃声**:磬声(默认)/ 仅震动;试听按钮(复用 `bell.dart`);
- **实时反馈**:"每天 N 次 · 本周共 M 次通知"(iOS 超 64 高亮警告,见 §5);
- **立即测试**:马上发一条通知,验证响铃 / 震动 / 静音行为是否符合预期;
- **设置帮助入口**:「让提醒准时响起」——进入双端设置引导页(§9),iOS / Android 各一版。

**通知本体**:标题=App 名 / "正念提醒",正文=用户文案,磬声,点按打开一个极简
「正念 / 打坐」页(可一键转打坐計時)。**尊重系统静音**——静音 / 勿扰时不出声只留横幅。

**权限首启**:开启功能时按序请求:通知权限 → (安卓)精确闹钟/电池不受限 → 引导。
每步都先用一句话说明"为什么需要",拒绝后给"稍后在设置开启"的返回路径,不硬堵。

## 8. 权限与平台实现清单

**新增依赖**:`flutter_local_notifications`、`timezone`;(可选)`device_info_plus`(识别厂商做引导)、
`android_intent_plus`(跳 OEM 设置)。**这套本地通知基建也是 PRD §5 通知中心 / 活动提醒(P2.1)所需**,一次投入两处受益。

**iOS**:
- `UNUserNotificationCenter` 申请 alert + sound 授权;
- 磬声打包进 bundle(`bell.caf` 或 `bell.wav`,**必须 ≤ 30 秒**),在通知里指定 `UNNotificationSound`;
- 无需后台模式;**不申请** Critical Alerts(拿不到,也不需要)。

**安卓**:
- `POST_NOTIFICATIONS`(13+ 运行时权限);
- 通知渠道 importance = HIGH,声音 = 磬声(`res/raw/bell`);**渠道不请求越过勿扰**(尊重静音决策);
- **精确 vs 非精确闹钟**(重要取舍):
  - 若"到点差几分钟无所谓" → 用**非精确**(inexact)重复,**免申请 `SCHEDULE_EXACT_ALARM`,Play 审核更顺**,且更省电、大陆厂商更少拦截。**推荐**。
  - 若必须卡在 :00/:30 精确 → 需 `USE_EXACT_ALARM`/`SCHEDULE_EXACT_ALARM`(12+),Play 会审查用途。
  - **建议:正念提醒用非精确即可**(允许系统批处理数分钟),规避审核与耗电风险;设置页文案说明"提醒时间为大致时点"。
- `RECEIVE_BOOT_COMPLETED`(重启后恢复,插件处理);
- **不使用前台服务**(我们靠系统排程而非常驻进程)——恰好避开最易被杀的模式;
- 电池优化豁免:`REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`(申请) + §9 厂商引导。

## 9. 设置帮助界面(双端引导)

一个「让提醒准时响起」帮助页,依当前平台展示对应指引。用 `device_info_plus` 识别平台 / 品牌;
每一步都配:一句"为什么需要" + 一个「去设置」按钮(尽力深链,失败退化为图文步骤) + 一个「已完成」勾选。
顶部放**「响铃自检」**:用户点一下发一条测试通知,自查是否真的响了。

### 9.1 iOS 指引
- **允许通知**(首启系统弹窗;若曾拒绝 → 深链「设置 > 善护念 > 通知」打开"允许通知""声音""锁屏显示");
- **静音 / 勿扰说明**:告知静音档 / 勿扰开启时不会出声(设计决策,非 Bug),需要出声请退出静音;
- **专注模式(Focus)**:如启用了"工作/睡眠"等专注,可能拦通知——指引把善护念加入允许列表;
- **(可选)后台 App 刷新**:仅当槽位 > 64 走滚动降级模式时才需要,用于后台补排;≤64 无需。

### 9.2 Android 通用指引
- **允许通知**(13+ 运行时权限;深链通知设置);
- **电池不受限**:引导关闭对本 App 的电池优化(标准 `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`);
- **勿扰说明**:同 iOS,系统勿扰时不出声。

### 9.3 大陆厂商专项(最大风险,1/3 用户)
MIUI / HarmonyOS(EMUI)/ ColorOS / OriginOS 等会杀后台并可能延迟 / 丢弃闹钟,除非用户**手动**放行:
- 按品牌给**分厂商图文步骤**:开启「自启动 / 自动启动」、「允许后台活动 / 无限制」、「锁定在最近任务」、「允许后台弹通知」;
- 尽力用 `android_intent` 深链到对应设置页(多为未公开 intent,失败则退化为纯图文);
- **诚实边界**:即便全部放行,个别厂商仍可能延迟或漏响——系统级限制,非本 App 缺陷;帮助页与设置页都写明,避免误解为 Bug。

## 10. 双端一致性与现有功能关系

### 10.1 双端一致性(用户决策 #9)
- **同一套**设置页 UI、文案、磬声、交互逻辑、通知呈现——Flutter 层共享,不为平台分叉。
- 唯一允许的差异是**系统强加**的,且以"提示信息"呈现、不做成两套功能:
  - iOS 槽位 > 64 的实时警告与降级(安卓无此提示);
  - 帮助页按平台显示对应步骤(§9);
  - "尊重系统静音"两端行为一致(iOS 无法穿透、安卓主动选择不穿透)。
- 调度差异(iOS 日历重复触发 / 安卓 zonedSchedule 周重复)封装在一个 `ReminderScheduler` 抽象后,
  上层无感知——保证行为一致、便于测试。

### 10.2 与现有功能关系

- **打坐計時「中途铃」(PRD §9.3 / timer_screen)= 前台版**正念提醒(App 开着、计时中每 X 分钟一声);
  本功能是它的 **OS 级、App 可关闭版**,共用磬声与"正念提醒"概念。当初 PLAN P2.6 已注明此项"推迟到需要本地通知基建时做"。
- **本地通知基建(`flutter_local_notifications`)与 P2.1 共享**:P2.1 的活动提醒、免打扰的本地兜底也要用它。
  建议本功能与 P2.1 的本地通知部分合并投入。
- **App 内免打扰时段(P2.1,默认 22:00–07:00)**:那是"不发**推送**"的应用级设置;
  正念提醒是用户**显式设定**的窗口,不受应用级免打扰约束(用户故意要在这个窗口被提醒);
  但**系统级静音 / 勿扰**仍然生效(§1 决策)。两者层级不同,需在实现时明确区分。

## 11. 边界情况

- **间隔与窗口不整除**:最后一次 ≤ end 即止(如 8:00–9:00 每 40 分 → 8:00、8:40)。
- **start ≥ end**:设置页校验拦截("结束时间需晚于开始时间");不支持跨午夜(§4.1)。
- **改配置**:任何字段变更 → 先 `cancelAll()` 本功能的通知(用固定 id 段)再全量重排。
- **关总开关**:取消本功能全部通知(用专属 id 区间,勿误删 P2.1 的活动通知)。
- **重启 / 换机**:安卓 BOOT_COMPLETED 恢复;iOS 重复通知本就跨重启存活;换机需重设(本地存储不迁移)。
- **权限被撤**:检测通知权限关闭时,设置页顶部提示"提醒已失效,点此重新授权"。

## 12. 测试计划

- **纯逻辑单测**:槽位展开函数(周几×时间窗×间隔 → 槽集合)、iOS 64 超限判定、不整除边界、start≥end 校验。
- **真机验证矩阵**(依赖 E6 大陆网络无关,但需真机):
  iOS(息屏 / 关 App / 静音开关 / 勿扰)、海外安卓、**大陆安卓至少一台厂商机**(小米 / 华为)验证放行前后差异。
- **回归**:确认不影响 P2.1 活动通知(id 区间隔离)。

## 13. 分期实施建议

1. **阶段 A(基建 + 安卓)**:引入 `flutter_local_notifications`/`timezone`;数据模型 + 槽位展开(带单测)+ 设置页 + 安卓排程 + 厂商引导 + 立即测试。安卓对"关 App 也能响"支持最好,先真机验证闭环。
2. **阶段 B(iOS)**:iOS 授权 + 64 槽策略 + 超限 UX;真机验证关 App / 静音行为。
3. **阶段 C(打磨)**:点击进正念页、与打坐計時互转、文案与简繁本地化、无障碍与大字号走查。
