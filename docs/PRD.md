## 1. 产品概述

**善护念 / PureThoughts** 是面向共修团体的功课记录与学修一体化 App。核心是**分群报数与统计**,并整合通知活动、直播回看、在线经本、念诵导引音视频、以及打坐工具。

- **目标用户**:全球华人共修社群成员,**约 1/3 在中国大陆**;**年龄跨度大**(需大字体、简洁交互)。
- **平台**:iOS + Android(Flutter 一套代码)。
- **语言**:简体 / 繁体,用户可切换,**默认繁体**。
- **文化基调**:随喜、不攀比 → **不做任何成员间排名**;个人数据仅自己可见,群只见总量。

## 3. 报数群模型(核心结构)

### 3.1 入群流程
1. 用户注册登录(**用户名 + 密码**,v0.5.9:面向年长/无邮箱用户,注册零门槛;邮箱为选填,仅用于找回密码)。
2. 输入目标群的 **群 ID**(群主分享),**并附一条申请说明**,提交申请。
3. 群主在"成员审核"列表中查看申请说明,通过 / 拒绝。
4. 通过后成为该群成员,即可报数。

### 3.2 群生命周期(v0.4 补)
- **退群**:成员可随时退群;历史报数记录保留(群总量不变),个人明细对本人仍可见。
- **移除成员**:群主可移除成员(状态 `removed`);数据处理同退群。被移除者可重新申请。
- **群主转让**:群主可将群主身份转让给群内任一 `approved` 成员;转让后自己降为普通成员。
- **解散群**:仅群主 / 管理员可解散;解散前需二次确认。群数据软删除保留(供管理员追溯),对成员不可见。
- **群 ID(join_code)重置**:群主可重新生成 join_code(旧码失效),用于群 ID 泄露后止损。
- **群公告**:群主可发一条置顶公告(如本期共修安排),群内成员可见;更新时生成一条 App 内通知。

**说明**
- **注册用户可自行建群,建群者自动成为群主**;管理员也可建群 / 指定群主。
- 群成员状态:`pending / approved / rejected / removed / left`。
- "共修整体统计" = 群内所有成员的汇总(含已退群成员的历史贡献)。

## 4. Epic 1 · 功课报数(核心)

### 4.1 功课清单
- **功课项必须具体**(v0.5.2 修正):經/咒/懺是**分类**而非功课项,不能笼统报"诵经 2 部"——报数对象是具体条目,如金剛經 1 部、地藏經 3 部、大悲咒 2 遍、地藏懺 1 遍。
- 每个功课项含:名称(简/繁双语)、**分类**(`經 / 咒 / 懺 / 念佛 / 靜坐 / 其他`)、**计数单位**;清单与报数选择器按分类分组展示,统计可按分类拆分。
- **主清单**由 App 管理员维护(全局);**用户可在群内加自定义功课项**(**群成员均可添加**,需选分类与单位;群主可停用或合并重复项,防清单杂乱)。
- **计数单位**:`部`(经)、`遍`(咒 / 忏)、`次`、`分钟`(打坐等);单位挂在具体条目上,同类条目可用不同单位。

### 4.2 报数
- 提交当日 / 当次功课数量。
- 一条报数记录含:群、报数人、**被报对象**、功课项、数量、单位、备注、报数时间。
- **代报**:用户可帮任何人报,**被报对象不限于群成员**。对象选择器三个来源合一:
  - ① **群成员**(计入其个人统计);
  - ② **本群代报名单**(之前用过的自由名字,点选即用);
  - ③ **直接输入任意新名字**(仅计入群总量,不绑账号)。
  - **自由名字记忆**(v0.5.1 补):输入过的自由名字自动存入**本群代报名单**,下次输入时自动补全 / 点选;名单**全群成员共享**(避免同一人被写成多种写法,群总量好归口);名单条目可由添加者或群主删除(处理输错的名字,已产生的报数记录不受影响)。
  - **代报通知**(v0.4 补):被代报的**群成员**会收到一条 App 内通知("某某为您报了 X");被代报人可**删除**该条记录(数据主权归本人)。自由名字对象无账号,不涉及通知。
- **补报("报之前的")**:随时补记忘记报的功课,**统一计入报数当天的统计**(不做历史日期归属);如需说明实际日期可写在备注。
- **报数交互定案**(v0.5.3,用户批准):
  - **批量多选**:一次报多种功课,每项独立数量(默认上次报的值,带 ± 步进按钮);
  - **首页「報數」直达**:单群直接进表单,多群弹选择器并记住上次所选;
  - **首页分组宫格**(v0.5.5):功能按语义分组呈现——**日課**(報數、快捷報數)/ **共修**(我的群組、活動日曆)/ **修行**(個人統計、我的發願、打坐計時、念珠計數)/ **通用**(通知、設定);日課组用大色块强调,布局大气简约、充分利用空间;通知与設定同时保留右上角小图标;
  - **快捷报数可多选**(v0.5.5):点开底部弹层,常用组合(可跨群)多选 + 核对各项数量(默认上次值,± 步进)后**一次提交**;
  - **「重複上次」**:一键带入上次(最近一天)自报的功课组合与数量;
  - **常用功课置顶**:选择区最前为「常用」组(按我最近报过排序),其余按分类分组;
  - **对象选择折叠**:默认报自己,点「替他人報數」才展开三来源选择器;
  - **提交栏固定底部**(显示已选项数);**提交成功展示随喜反馈 + 本次摘要**。
  - **报数记录按批次分组显示**(v0.5.10):**同一次提交**的多条功课**一起显示为一张卡片**(卡头为对象·日期·代报人·备注,卡内逐条列功课与数量),不再一条一条平铺;批次判定 = 同一报数人 + 同一对象 + 同一提交时刻(`created_at`,单条多行 insert 的事务时间);修改 / 删除 / 举报仍按单条(log id)生效。应用于群报数记录页与个人统计的历史明细。
- **修改 / 删除**(v0.4 补):
  - 报数人可修改 / 删除**自己提交的**记录(含代报出去的);
  - 被代报人可删除**以自己为对象**的记录;
  - 群主可删除本群任意记录(处理恶意灌水);
  - 删除为**软删除**(`deleted_at`),统计即时扣减,后台可审计。
- **快捷报数**:首页常用功课一键报数;记住每个功课上次填的数量作默认值(老年用户友好)。

### 4.3 统计与 Dashboard
- **只显示个人数据与群总量,不做任何排名 / 对比。**
- **统计时区口径**(v0.4 明确):每条报数落库时按**报数人当时的本地时区**计算出 `local_date`(自然日),所有"今日 / 按日趋势"统计一律以 `local_date` 聚合 —— 个人与群共用同一口径,群的"今日" = 各成员"自己的今天"之和。
- Dashboard 指标:
  - 个人:今日 / 累计,按功课类型拆分。
  - **个人连续用功天数**:连续有报数的自然日(按 `local_date`),**仅自己可见**、不进群统计;中断只温和归零,不做"断签"提醒(守住不攀比、不施压的基调)。
  - 群整体:今日 / 累计。
  - **近 N 天趋势图**。
  - **发愿目标进度**(见 4.4)。
- **历史查看**:按日期 / 功课类型回顾过往报数。
- **群主视角**:群主只看群总量与"今日已报人数"等聚合指标,**不提供成员明细排行**(守住不攀比基调);但审核 / 管理需要时可查单个成员的记录列表。
- **数据导出**:本轮不做(留作备选:群主导出时间段汇总 CSV,用于法会回向名单)。

### 4.4 发愿 / 立目标
- 用户可发愿:某功课在某周期内念满目标数量(如"90 天内念咒 10 万遍")。
- **进度归属**(v0.4 明确):发愿默认统计该用户**所有群**中该功课项在周期内的报数总和(含别人代报给自己的);也可在发愿时限定单个群。
- 发愿状态:`active / completed / expired / abandoned`;到期未完成不提示"失败",仅温和显示进度(文化基调)。
- Dashboard 展示**目标进度条**(已完成 / 目标、剩余天数);**达成时显示随喜祝贺页 + 回向文**(回向文文案由内容方提供,存后台可配置)。
- 发愿仅本人可见,不在群内公示(不攀比)。

## 5. Epic 2 · 通知 / 活动 / 日历

- **循环模板 + 单次修改**:周六共修、周三打坐等按循环规则自动生成,单次可改时间/内容/取消。
- **通知类型**:周六共修预告(提前一天)、周六当天 Webex/YouTube 连接、周三打坐提醒、讲法/共修/法会预告。
- **跨时区**:全球用户 → 活动时间以 `timestamptz` 存储,**按用户本地时区显示**。
- **日历视图**:所有活动可在日历查看;**未来事件列表**(v0.5.7:日历下方按时间列出即将到来的活动);同一天多种活动共存并全部显示。
- **事件类型**(v0.5.7 定案):动态表 `event_types`,默认 **靜坐 / 共修 / 講法 / 禪七 / 其它**;**管理员可增删改类型**(名称简繁 + 图标 + 启用状态,图标从预置图标集选择);不同类型在日历中显示不同图标;被活动引用的类型不可删、只能停用。
- **管理员活动管理**(v0.5.7):新增 / 编辑 / 删除整个活动(含循环全部场次)/ 取消单次;**任何活动变更自动生成全员通知**(新增/更新/删除/单次取消,App 内通知中心即时可见,推送接通后同链路升级)。**新建活动默认不重复**(v0.5.10:「每週重複」开关默认关闭,单次活动为主;需要每周循环时管理员手动开启)。
- **活动时间表与相关资料**(v0.5.12 定案,详细设计见 [`design/event-agenda.md`](design/event-agenda.md)):管理员可给活动补一份**时间表**(几点到几点做什么,**支持跨多天**如禪七)与**相关资料**;用户可看、可**整张时间表分享/复制**转发 Line/微信。
  - **时间表**:结构化行(第几天 + 起讫时间 + 活动),每行可选填一个**自由网址**(如读经行链接到经文网页);归属活动本身,循环活动各场次共用;时间为现场墙钟时间原样显示、不做时区换算。
  - **相关资料**:管理员**在 App 内上传 PDF**(如经本)供用户下载 —— 存 **Supabase Storage**(`event-files` bucket,公开可读、仅管理员可写);走自有域名 `api.pure-thoughts.com`,**大陆可达**(相对 YouTube 的关键优势)。另沿用活动级 YouTube/Webex 链接。
  - **分享**:整张时间表渲染为纯文本(含各行链接、PDF 公开 URL、YouTube),经系统分享面板发 Line/微信,或复制到剪贴板;转发后大陆用户点 Storage 链接可下 PDF。
  - 均**管理员维护、用户只读**(匿名可看);改时间表/资料不额外发全员通知。
- **分类订阅**:用户可按类型开关推送。
- **免打扰时段**:默认 22:00–07:00(用户本地时区)不发系统推送,顺延到时段结束后发;活动开始前的实时通知(如"共修连接")不受限;用户可在设置中调整或关闭。

### 5.1 推送策略(重要:约 1/3 用户在中国大陆)

Supabase 不发推送,由 Edge Function / DB 触发外部通道。**不接国内厂商推送**,分三层:

| 通道 | 覆盖 | 说明 |
|---|---|---|
| **APNs(原生,不经 FCM)** | 全部 iOS(含大陆) | APNs 在大陆可用;不走 FCM 以免注册拿不到 token |
| **FCM** | 海外 Android | 大陆 Android 收不到(依赖 Google 服务) |
| **App 内通知中心(拉取式)** | **全部,含大陆 Android** | 通知存 Supabase,开 App 即拉取 + 红点角标 |

- ⚠️ **关键后果**:**大陆 Android 用户收不到实时系统推送**,只能靠"打开 App 即见"→ 因此 **App 内通知中心是刚需,不是增强项**。
- **邮件兜底**:时效性强的通知(如周六共修预告)对大陆用户额外发邮件(经 Resend / SES 等,由 Edge Function 触发)。
  - **如何识别大陆用户**(v0.4 补):注册 / 设置中让用户自选"所在地区"(大陆 / 其他);另对 **FCM token 注册失败的 Android 设备自动标记**为需邮件兜底。
- 未来若确需大陆 Android 实时推送,再评估接厂商通道(当前明确不接)。

## 6. Epic 3 · 在线视频与往期录制

- **固定频道**(v0.5.6 定案):YouTube `youtube.com/@善護念`;Webex `purethoughts.my.webex.com/join/Shanhunian`;日历活动的直播链接默认预填这两个频道。
- **开播自动检测**(v0.5.6):`live-probe` Edge Function 探测 YouTube 频道 `/live` 端点(无需 API key)→ 开播时写 `live_streams` 表并生成全员通知(App 内通知中心即时可见;P2.1 推送接通后同链路升级系统推送);打开直播页时客户端触发探测,生产部署后 pg_cron 每 5 分钟服务端探测。**Webex 无公开状态 API**,不做自动检测:固定房间一键加入 + 日历活动预告为准。
- **直播入口**:YouTube / Webex。
- **播放方式**:**YouTube 内嵌播放**;**Webex 应用内 WebView 加入**(v0.5.7 定案:媒体权限桥接支持网页版通话;App 显示名尽力预填访客名,失效则手动填、WebView 会记住;**所有 Webex 入口永远保留「用 Webex App 開啟」选项**——网页版兼容性风险的兜底)。
- **往期回看**:YouTube 存储的录制视频。
- **讲法问答检索**(v0.5.11 定案,详细设计见 [`design/qa-search.md`](design/qa-search.md)):对接内容方已有的问答检索 API
  (`www.pure-thoughts.com/api`,FastAPI;一条记录 = 一个**问答片段**,含 AI 摘要正文 + 带时间戳的 YouTube 链接)。
  - **客户端直连,不走 `qa-proxy` 代理**:该接口完全公开、无认证、CORS 全开,代理的两个立项理由(隐藏上游地址、统一鉴权)均不成立;直连少一跳、少一个故障点,且问答与 Supabase 无数据关系,不该被它拖挂。基址常量在 `core/channels.dart`,日后若需保护再引代理只改一处。
  - **简繁归一化在后端做**:库内简繁混杂(实测 `禅修`→6 条、`禪修`→1 条,互不包含),而 App 默认繁体 —— 不治理则搜索对默认语言用户基本不可用。方案:入库时 OpenCC 生成 hans/hant 双字形 + 简体归一检索列,`/search` 与 `/tags` 加 `script=hans|hant` 参数(响应字段名不变)。**此为客户端实现的硬前置**(PLAN E14)。
  - **详情页以播放器为主、摘要正文在下**;受 YouTube 嵌入限制(错误 153,真机已验),播放形态为「封面 + 大播放按钮 → 全屏 `/watch` 页并跳到该问题的时间戳」,非内嵌 iframe。
  - **摘要正文永远完整渲染**:大陆用户播不了 YouTube,但接口与摘要可达 —— 摘要是他们能拿到的全部价值,不得因播放失败而缺失(§14.2)。
  - 标签用**可搜索的选择器**(归一后约 600 个,全量 chips 不可用);多标签为「或」关系。匿名可用。
- ⚠️ 大陆可达性风险见 §14。

### 12.3 RLS 策略要点(每表一句话)

| 表 | 策略 |
|---|---|
| `profiles` | 本人可读写自己;群主可读本群成员的 display_name(经视图收敛字段);管理员全读 |
| `groups` | approved 成员可读;owner 可更新;**join_code 不直接可读**,入群走 RPC(`join_group(code, message)`)校验,防遍历 |
| `group_members` | 本人可读自己的记录;群主可读写本群全部;insert 仅限本人申请 |
| `practice_logs` | 本群 approved 成员可读(`deleted_at is null`);insert 限成员;update/delete 限 reporter 本人、subject 本人(仅删)、群主 |
| `proxy_names` | 本群 approved 成员可读、可 insert(报数时自动记入);删除限添加者 / 群主 |
| `vows` | 仅本人读写 |
| `practice_types` | 全局项所有人可读;群自定义项限本群成员读、群主写 |
| `events` / `media_items` / `scriptures` | **anon 可读**(匿名浏览),仅管理员写 |
| `notifications` | scope 命中者可读;仅服务端(service_role)写 |
| `reports` | 举报人可 insert / 读自己的;管理员全权 |
| `user_blocks` | 仅本人读写 |

### 12.4 Edge Functions 与定时任务

| 函数 | 触发 | 职责 |
|---|---|---|
| `push-dispatch` | DB webhook(notifications insert)/ cron | 查目标用户 token,APNs(JWT 直连)+ FCM 发送;失败 token 标记清理 |
| `email-fallback` | 由 push-dispatch 级联 | 对 region=cn 或 fcm_failed 的用户发邮件(Resend/SES) |
| `delete-account` | 客户端调用(需登录) | §10.1 删除 + 匿名化流程 |

> ~~`qa-proxy`~~ **已取消**(v0.5.11):上游问答 API 完全公开、无认证、CORS 全开,「隐藏上游地址 / 统一鉴权」两个立项理由均不成立;客户端直连(§6)。

**pg_cron**:每日滚动展开未来 14 天活动 occurrence(应用 `event_overrides`)→ 生成 `notifications`(scheduled_at = 规则时间,如提前一天)→ 到点由 cron 触发 `push-dispatch` 投递。

### 12.5 自托管部署要点(v0.5 新增)

Supabase 全套开源,自托管(官方 Docker Compose,或 Coolify 等面板)与云端 **SDK 完全兼容**:`supabase_flutter` 只需指向自有域名 + anon key,本章 RLS / Edge Functions / pg_cron 设计不变。**结论:可行,且机房选址可直接缓解大陆可达性风险**;代价是运维责任全部自担。

- **机房选址**:⚠️ **不要放中国大陆境内**(需 ICP 备案,且宗教内容境内托管有合规风险)。选大陆直连质量较好的境外机房:**香港 / 新加坡 / 东京**;上线前用大陆真实网络实测延迟与丢包后定。
- **暴露面**:只暴露 Kong 网关(443 + TLS,自有域名);Postgres、Studio 不上公网(Studio 走内网 / VPN / IP 白名单)。
- **Auth 自配项**:**SMTP**(重置密码邮件、§5.1 邮件兜底;v0.5.9 起注册不再依赖邮件,SMTP 非上线硬阻塞)——可用 Resend / SES。Google/Apple OAuth 已移出 MVP,将来如加回再配回调域名与密钥。
- **Edge Functions**:自托管 `edge-runtime` 容器运行 §12.4 四个函数;若维护不便,可退化为同机的一个小型 Node/Deno 服务 + 反向代理,职责不变。
- **pg_cron / pg_net**:官方 Postgres 镜像已含扩展,启用即可,排程链路不变。
- **备份与恢复**:每日全量 + WAL 归档到**异地**对象存储(B2/R2/S3);每季度恢复演练——自托管没有云端 PITR 兜底,"备份可恢复"是第一优先级。
- **升级与监控**:跟进 Supabase 镜像安全更新(先在测试环境验证再升级生产);Uptime 监控 + 资源告警。本 App 量级(数百至数千用户)单机足够,先保证备份,再考虑高可用。
