## 8. 使用规则(计划 → 开发 → 跟踪 → 测试)

1. **开工**:先读 §1 进度总览 → 进入当前 Phase,按任务编号顺序领取未勾选任务(标注"可并行"的除外);检查该任务是否被 §2 外部依赖阻塞。
2. **开发**:每个任务开 feature 分支(如 `feature/p1-6-practice-log`);commit 信息以任务编号开头(如 `P1.6: 代报三来源选择器`);遵循 CLAUDE.md 已定案决策,不重新讨论。
3. **跟踪**:任务完成(含其验收项)→ 勾选 checkbox → 更新 §1 该 Phase 的进度计数;Phase 内任务全完成且 **DoD 通过** → 状态改 ✅ 并填完成日期;遇阻塞 → 任务标 ⛔ 并登记到 §7。
4. **测试**:每个任务行内写了各自验收;Phase DoD 是升级门槛,不达标不得进入下一 Phase;代码任务完成的底线是 `flutter analyze` + `flutter test` 全绿(P0.8 之后强制)。
5. **变更**:新需求 → 先改 PRD(升版本号)→ 再把任务加进本文对应 Phase(编号顺延)→ 才开始实现。只在本文加任务、不改 PRD 的,视为无效需求。
6. **本文维护**:每次更新同时刷新顶部"最后更新"日期;Phase 完成时在 §1 填日期,作为简易里程碑记录。
